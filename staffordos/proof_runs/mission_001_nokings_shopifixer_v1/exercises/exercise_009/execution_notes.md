## Footer Content Ownership

Owning section: `sections/footer.liquid`.

Repository-backed layout engine:

- Renders configured blocks through `{% content_for 'blocks' %}` inside `.footer-content`.
- Grid columns: `total_blocks | at_most: 4`.
- Orphan handling: computes `last_row_count = (total_blocks - 1) % 4 + 1` and `tablet_last_row_count = (total_blocks - 1) % 2 + 1`; when the last row would hold a single item (`total_blocks > 4` with `last_row_count == 1` on desktop, or exactly 5 blocks on tablet), the last child spans the full row (`grid-column: 1 / -1`).
- Responsive behavior: single column below 750px; up to 3 columns 750-989px (4-column configs collapse to 2); `--grid-columns` columns at 990px+; a single block centers.
- Performance containment: `contain: content; content-visibility: auto`.
- Styling inputs: `render 'spacing-style'`, `--footer-gap`, section `color_scheme`, `section_width`.

Archived configured content blocks:

- One `group` block (column direction, vertical on mobile) containing two `text` blocks: `<h2>Join our email list</h2>` and `<p>Get exclusive deals and early access to new products.</p>`.
- One `email-signup` block (label `Sign up`, integrated arrow button, border radius 100).

## Footer Utility Ownership

Owning section: `sections/footer-utilities.liquid`.

Repository-backed facts:

- Renders configured blocks through `{% content_for 'blocks' %}` inside `.utilities`, marked `data-testid="footer-utilities"`.
- Schema: `max_blocks: 3`, `enabled_on: { groups: ["footer"] }` — the utilities section is footer-group-exclusive by schema.
- Layout: flex column centered below 750px; at 750px+ a three-slot grid (`1fr auto 1fr`) with per-count positioning CSS for 1, 2, or 3 blocks (start / start+end / start+center+end).
- Divider: optional top border driven by `--border-width` from `divider_thickness` (archived value `0`).
- Utility text uses `--color-foreground-muted`.

Archived configured utility blocks, in order:

## Signup Flow

Repository-backed signup path:

`footer-group.json` email-signup block
-> `blocks/email-signup.liquid`
-> `{% form 'customer' %}`
-> Shopify platform form handling

Repository-backed facts:

- Input is `name="contact[email]"` with `type="email"`, `required`, autocomplete/autocorrect handling, and a visually hidden label from `blocks.email_signup.label`.
- Error state: `form.errors` drives `autofocus`, `aria-invalid`, and an `aria-describedby` error message region.
- Success state: `form.posted_successfully?` drives a success message region.
- Submit affordance: text label or arrow icon (`render 'icon', icon: 'arrow'`) per `display_type`; archived config uses the integrated arrow style.
- Style variants: integrated button, underline, no-border, arrow, optional non-inherited color scheme.

Platform boundary:

- `{% form 'customer' %}` posts to Shopify's platform customer endpoint. Submission handling, customer-list writes, consent state, and any email delivery are platform runtime behavior — Not Yet Proven.

## Reusable ShopiFixer Patterns

- Footer architecture splits into a content section and a utilities section; inventory each half separately.
- Start footer inventory at the auto-generated group JSON, then trace sections, then blocks — same discipline as header (Exercise 007).
- Distinguish schema-supported capability from configured content before recommending any footer change.
- Treat `{% form 'customer' %}`, `shop.policies`, and `shop.enabled_payment_types` as platform boundaries: source proves the rendering path, never the data.
- Grid orphan-handling logic means footer block count changes have layout side effects — a one-block addition can re-flow the entire footer grid.
- Utility rows with `max_blocks` and per-count CSS mean utility-block changes are positional, not just additive.
- Platform-root default social URLs are theme-preset artifacts, not merchant presence.

## Risks

- Treating schema-supported footer blocks (menu, payment-icons, @app) as configured content would overstate archived truth.
- Treating rendered policy links as merchant policy claims would cross the merchant-authorship boundary.
- Treating the email-signup block as a working subscription flow would claim platform runtime behavior.
- Treating platform-root social URLs as live merchant social presence would overclaim.
- Treating archived group JSON as current live footer configuration ignores admin/theme-editor drift.
- Footer block-count changes interact with the orphan-handling grid logic; naive additions can change layout on desktop and tablet simultaneously.
- Utility-row changes are constrained by `max_blocks: 3` and per-count positioning CSS.
