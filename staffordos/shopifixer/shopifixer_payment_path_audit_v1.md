# ShopiFixer Payment Path Audit V1

## Verdict

The path is **partial**. A real merchant can reach an offer and a checkout entry point, and the codebase contains Stripe and client-registry payment writers, but the current ShopiFixer flow does not bridge payment into client truth and then into fulfillment truth automatically.

## Current State

`offer_sent`

## Target State

`payment_received`

## Checkout Path

The ShopiFixer page posts to:

- `/__public-checkout`
- or `${NEXT_PUBLIC_ABANDO_API_BASE || NEXT_PUBLIC_ABANDO_BACKEND_ORIGIN}/__public-checkout`

The route itself is not present in the checked-in repo, so the exact live handler cannot be verified here.

## Payment Authority

The payment authority exists in Stripe-backed code:

- `abando-frontend/src/lib/stripe.ts`
- `abando-frontend/src/app_legacy/api/trial/link/route.ts`
- `abando-frontend/src/app_legacy/api/stripe/webhook/route.ts`
- `staffordos/clients/client_registry_v1.mjs`
- `staffordos/revenue/revenue_agent_v1.mjs`

The successful payment event is `checkout.session.completed`.

The actual truth writer is:

- `staffordos/clients/client_registry_v1.mjs:recordVerifiedStripePayment`

The propagation writer is:

- `staffordos/revenue/revenue_agent_v1.mjs:recordStripePaymentPropagation`

## Fulfillment Trigger

The fulfillment read model is generated by:

- `staffordos/fulfillment/build_shopifixer_fulfillment_truth_v1.mjs`

That file is an offline builder. No runtime bridge was found that updates fulfillment truth after payment.

## Runtime Flow Map

1. Authorized offer exists and is send-ready.
2. The ShopiFixer page posts to `/__public-checkout`.
3. Stripe payment can be created/confirmed in legacy or external checkout code.
4. `checkout.session.completed` is the verified payment event.
5. `recordVerifiedStripePayment` can write paid truth into `client_registry_v1.json`.
6. `recordStripePaymentPropagation` can also update revenue truth.
7. The fulfillment builder can represent a paid sprint as `payment_received`.
8. The current ShopiFixer path does not connect steps 3-7 end to end.

## Runtime Blockers

- No in-repo handler for `/__public-checkout` was found.
- The current ShopiFixer checkout flow does not call the verified-payment writer.
- The legacy Stripe webhook route only issues a magic link and does not record ShopiFixer payment truth.
- The fulfillment truth builder is offline only.
- `shopifixer_fulfillment_truth_v1.json` remains `waiting_for_payment`.

## Readiness Score

`40/100`

This is not a sendability score. It is a runtime path score. The offer and send authorities are ready, but the post-payment bridge is missing.

## Smallest Repair

Wire the ShopiFixer checkout completion path to the existing payment propagation writer, then rebuild the fulfillment truth from the paid client record.

## Exact Next Repair

1. Add the missing runtime bridge from the ShopiFixer payment event to `recordStripePaymentPropagation`.
2. Pass the same merchant identity through the payment payload.
3. Rebuild `staffordos/fulfillment/shopifixer_fulfillment_truth_v1.json` from the updated client truth.

## What Becomes Provable After Payment

- the offer was actually paid
- the client registry contains verified payment truth
- the fulfillment item can move to `payment_received`
- the proof-backed fulfillment loop can begin

## Summary

The codebase can already represent payment truth and fulfillment truth, but the current ShopiFixer runtime path does not connect them.
