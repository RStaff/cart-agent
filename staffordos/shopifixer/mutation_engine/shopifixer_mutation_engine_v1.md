# SHOPIFIXER MUTATION ENGINE V1

## Purpose

Define how ShopiFixer moves from audit to safe website improvement.

ShopiFixer should not blindly mutate websites.

ShopiFixer should:

1. inspect
2. fingerprint
3. diagnose
4. match pattern
5. map to code/theme
6. generate safe change plan
7. execute scoped fix
8. QA
9. capture proof
10. learn

## Current Focus

Shopify first.

Do not generalize to every website platform until Shopify sprint delivery is proven.

## Core Question

What is the smallest safe website change that can improve revenue intake probability and be proven visually?

## Required Inputs

- desktop screenshot
- mobile screenshot
- store URL
- Shopify theme source
- templates/index.json
- relevant section files
- product data
- CTA destinations
- trust claims
- existing ShopiFixer pattern library

## Required Output

For each store, produce:

- detected issue
- matched pattern
- affected theme section
- affected file or admin area
- implementation level
- minimal safe change
- risk level
- QA checklist
- before/after proof plan
- merchant-facing explanation

## Mutation Levels

Level 1:
Settings-only change.

Level 2:
Theme editor section reorder, product binding, CTA/copy/trust placement.

Level 3:
Small custom Liquid section.

Level 4:
Scoped code change across section/snippet/CSS.

Level 5:
Custom project. Not included in standard $950 sprint.

## $950 Sprint Boundary

A standard ShopiFixer sprint should usually stay within Level 2 or Level 3.

Level 4 is allowed only when:

- one visible issue is being fixed
- affected files are known
- rollback is clear
- QA is simple
- proof is visual

Level 5 requires custom quote.

## What Makes This Premium

ShopiFixer is not selling quick edits.

ShopiFixer is selling:

- diagnosis
- code/theme understanding
- safe implementation
- QA
- visible proof
- repeatable learning

## Speed Strategy

Speed comes from:

- reusable patterns
- theme fingerprints
- known section structures
- prebuilt QA checklists
- proof package templates

Speed does not come from skipping inspection.

## Safety Rules

Never mutate before:

- theme backup exists
- affected section is identified
- rollback path is known
- product data is verified
- claims are verified
- QA checklist exists

## Learning Rule

Every completed sprint must create a learning record:

- store
- theme
- pattern
- implementation level
- files touched
- time spent
- QA result
- proof package created
- merchant outcome
- revenue outcome

## Expansion Rule

Do not market ShopiFixer as an any-website mutator yet.

Internal long-term path:

1. Shopify
2. WordPress / WooCommerce
3. Webflow
4. Custom React/Next.js sites
5. General codebase mutation

Only expand after Shopify proof is repeatable.

## Success Standard

ShopiFixer becomes one-of-a-kind when it can repeatedly answer:

- what is wrong?
- where does it live?
- what should change?
- how risky is the change?
- how fast can it be done?
- how will proof be captured?
- what did we learn for the next sprint?
