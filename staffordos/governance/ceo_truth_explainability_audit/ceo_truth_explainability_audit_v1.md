# CEO Truth Explainability Audit V1

Generated at: 2026-06-08T22:17:11Z

## Verdict

The StaffordOS CEO Truth Snapshot is partially explainable and partially authoritative.

The primary action itself is authoritative because `staffordos/cockpit/ceo_truth_snapshot_v1.json` points at `staffordos/snapshots/primary_action_snapshot_v1.json`, which is generated by `staffordos/decision/resolve_primary_action_v1.mjs`.

The full CEO cockpit is not yet fully explainable because several displayed fields are local aggregations, heuristic blocker strings, or unavailable nulls without a field-level source map. The top-5 action list is explicitly synthesized inside `staffordos/cockpit/build_ceo_truth_snapshot_v1.mjs` and does not yet have a single governed top-5 action authority artifact.

There is also a builder/snapshot drift: the current `ceo_truth_snapshot_v1.json` contains primary-action execution fields such as `execution_status`, `last_launched_at`, `last_failed_at`, and `last_execution_result`, but the inspected builder source does not currently emit those fields from `primary_action_snapshot_v1.json`. Rerunning the builder could drop that action execution state from the CEO snapshot.

## Fields Not Traceable To A Single Authority

- `operator_actions.top_5_actions`: no governed top-5 authority artifact exists; the builder synthesizes and ranks candidates locally.
- `fulfillment.proof_needed`, `fulfillment.review_needed`, `fulfillment.referral_needed`: currently hard-coded as `null`; the snapshot itself warns these are not represented in runtime truth.
- `system_health.stale_truth.status`: set to `"unknown"` because no governed staleness threshold exists.
- `system_health.blockers[]`: explainable as builder output, but each blocker lacks a structured field-level trace object.
- `system_health.warnings[]`: explainable as builder output, but not authoritative business truth.
- `operator_actions.primary_action.context.revenue_gap`: reads `operator_dashboard_snapshot_v1.json: revenue_gaps[0].gap`; this assumes the first gap is the relevant primary-action gap.
- `operator_actions.primary_action.execution_*` in the current JSON: traceable to `primary_action_snapshot_v1.json` now, but not emitted by the inspected builder source.

## Exact Primary Action Logic

The CEO snapshot does not select the primary action directly. It imports it from:

`staffordos/snapshots/primary_action_snapshot_v1.json`

That file is generated by:

`staffordos/decision/resolve_primary_action_v1.mjs`

The resolver reads:

- `staffordos/snapshots/unit_work_snapshot_v1.json`
- `staffordos/clients/operator_dashboard_snapshot_v1.json`
- `staffordos/clients/client_registry_v1.json`
- `staffordos/gates/confidence_gate_v1.json`
- `staffordos/ux_audit/output/operator_command_center_ux_integrity_v1.json`

The winner is selected from `unit_work_snapshot_v1.json: open_work[]`.

Each open unit receives this score:

| Condition | Score |
|---|---:|
| `type == "opportunity"` and `stage` includes `"proposal"` | +45 |
| `type == "action"` and `next_action` includes `"follow up"` | +35 |
| `type == "delivery"` and `status == "waiting_for_payment"` | +25 |
| `domain_id == "shopifixer"` | +18 |
| `domain_id == "abando"` | +14 |
| `domain_id == "internal_dev"` | +6 |
| `owner == "ross"` | +10 |
| `next_action` includes `"close payment"` | +20 |
| `next_action` includes `"follow up"` | +12 |
| `status == "open"` | +5 |
| `status == "waiting_for_payment"` | +8 |

The score is capped at 100. Candidates are sorted by descending `priority_score`; the first candidate becomes `primary_action`.

For the current snapshot, `opp_cart-agent-dev.myshopify.com` wins because it is an open ShopiFixer opportunity in proposal stage, owned by Ross, and its next action explicitly says to follow up and close payment. Its raw score is 110 and is capped to 100.

## Truth Changes That Would Select A Different Primary Action

A different primary action would be selected if `staffordos/snapshots/unit_work_snapshot_v1.json: open_work[]` changed so another unit outranked or preceded the current winner.

Concrete changes:

- Remove `opp_cart-agent-dev.myshopify.com` from `open_work[]`.
- Change its `type` away from `"opportunity"`.
- Change its `stage` so it no longer includes `"proposal"`.
- Change its `domain_id` away from `"shopifixer"`.
- Change its `owner` away from `"ross"`.
- Change its `next_action` so it no longer includes `"follow up"` or `"close payment"`.
- Change its `status` away from `"open"`.
- Add a higher-scoring action to `open_work[]`. Because scores are capped at 100, an equal 100-point candidate can win if it appears before the current winner in the sorted candidate order.
- If no open work exists, the resolver emits `"No active unit of work found."`

Changes in client registry, revenue truth, dashboard snapshot, confidence gate, or UX report can change evidence, context, confidence band/risk wording, or fallback labels, but the inspected resolver's primary winner selection is driven by `unit_work_snapshot_v1.json: open_work[]`.

## Can The Cockpit Explain Why?

Partially.

The data can explain why through:

- `operator_actions.primary_action.evidence[]`
- `operator_actions.primary_action.risk[]`
- `operator_actions.primary_action.source_file`
- `operator_actions.top_5_actions.items[0].evidence[]`

The UI at `staffordos/ui/operator-frontend/app/operator/cockpit/page.tsx` displays the primary action, source, confidence, next step, expected outcome, and top-5 action cards with evidence. It does not currently display a structured scoring trace showing why this candidate beat each alternative.
